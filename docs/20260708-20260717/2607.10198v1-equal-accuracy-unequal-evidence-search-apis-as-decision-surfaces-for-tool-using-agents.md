## 摘要
搜索API是许多代理的基本检索层，往往也是它们最常使用的工具。传统搜索API提供URL、标题和预览网页内容的片段。由于全页面检索消耗大量token，代理检索架构越来越多地采用渐进式展示：代理首先看到片段，然后决定是否获取完整页面。在这类系统中，搜索API的性能通常主要根据答案准确度来评估。我们认为，商业搜索API更适合理解为一种决策面：由排序的片段、URL和元数据决定代理是立即回答、再次搜索还是花费token打开页面。我们使用一个冻结的GPT-5.4代理、两个工具（search_web和fetch_page）以及来自SEALQA-HARD的100个问题来验证这一观点，仅改变搜索提供商（Brave、Tavily、Firecrawl）。一个Kimi-K2.6预言机对代理可见的每个内容元素（URL、标题、片段以及获取的页面，若已获取）进行标注，产生了6,869个有效的逐URL判断。我们使用了经过审计的正确答案标签——语义匹配，它保留精确匹配，同时接受无害的格式和命名变体。在此衡量标准下，三个提供商的得分接近（分别为25、25、26/100），但它们的证据经济存在显著差异：Brave提供富含黄金答案的片段，Tavily将支持黄金答案的URL集中在排名第一的位置，而Firecrawl在这项固定代理策略下关联于更广泛的探索。我们还引入了表面矛盾与黄金URL的比率，该比率从0.92到2.59不等。因此，提供商的选择是一项检索预算和策略决策，而不仅仅是召回决策。

## Abstract
Search APIs are the fundamental retrieval layer for many agents and are often their most frequently used tool. Traditional search APIs provide URLs, titles, and snippets that preview website contents. Because full-page retrieval is token-intensive, agent retrieval architectures increasingly use progressive disclosure: the agent first sees snippets and then chooses whether to fetch full pages. In such systems, search API performance is often evaluated primarily by answer accuracy. We argue that a commercial search API is better understood as a decision surface: the ranked snippets, URLs, and metadata that determine whether an agent answers immediately, searches again, or spends tokens opening pages. We test this claim with one frozen GPT-5.4 agent, two tools (search_web and fetch_page), and 100 questions from SEALQA-HARD, varying only the search provider (Brave, Tavily, Firecrawl). A Kimi-K2.6 oracle labels every content element visible to the agent (URL, title, snippet, and fetched page, when fetched), producing 6,869 valid per-URL judgments. We use an audited correct-answer label, semantic match, which preserves exact matches while accepting harmless formatting and naming variants. Under this measure, the providers remain close (25, 25, 26 / 100), but their evidence economies differ sharply: Brave offers gold-answer-rich snippets, Tavily concentrates gold-supporting URLs at rank 1, and Firecrawl is associated with broader exploration under this fixed agent policy. We also introduce a surface contradiction-to-gold URL ratio, which varies from 0.92 to 2.59. Provider choice is therefore a retrieval-budget and policy decision, not merely a recall decision.

### 论文方法论
- **核心思想：将搜索API定义为“决策面”**
    - 论文提出，搜索API在代理看到任何页面全文之前，提供的排序URL、标题和片段构成的“预获取表面”，决定了代理是直接回答、再次搜索还是消耗token打开页面。评估应围绕这个表面的信息质量、误导性和行动引导性展开。
- **关键技术细节：冻结代理与逐URL预言机（Oracle）**
    - **冻结代理（Frozen Agent）**：整个实验中，回答模型（GPT-5.4）、提示词（Prompt）、工具集（`search_web` 和 `fetch_page`）、最大迭代次数（10次）和页面获取后端（Jina Reader）均被固定。唯一的实验变量是`search_web`工具所调用的商业搜索API提供商（Brave, Tavily, Firecrawl）。
    - **逐URL预言机标注协议**：引入一个Kimi-K2.6模型作为预言机，对代理轨迹中每个可见的URL元素（包括片段和已获取的页面全文）进行逐条标注。判断其是否包含黄金答案（`contains_gold_answer`）、是否支持模型答案、是否与黄金答案矛盾（`contradicts_gold_answer`）等，产生了6,869个有效判断。该预言机经过人工验证，一致率达94%。
    - **语义匹配**：正确答案采用经过审计的“语义匹配”标签，接受不影响语义的格式和命名变体，避免精确匹配的误判。
- **关键指标定义**：
    - **预获取表面支持（Pre-fetch Surface Support）**：黄金证据在片段或元数据中就可见的情况。
    - **决策四象限**：结合预获取支持的存在与否和代理是否获取了页面，将每条查询轨迹划分为`SMART`（有支持且已获取）、`MISSED`（有支持但未获取）、`BLIND`（无支持但进行了探索性获取）和`NO-OP`（无支持且未获取）。
    - **表面矛盾-黄金比率（Surface Contradiction-to-Gold Ratio, rc:g）**：仅在片段中，标记为“矛盾”的URL数与标记为“包含黄金答案”的URL数之比，用于衡量预获取表面的“污染”程度。

### 主要结论与发现
1.  **准确率等价，但证据经济迥异**：在语义匹配标准下，三大API的最终准确率非常接近（25、25、26/100），但其内部的证据供给和代理行为模式差异显著。
2.  **提供商呈现出不同的“决策面”画像**：
    - **Brave**：预获取支持最丰富（30个查询），片段富含黄金答案，但代理常因已有足量信息而放弃获取页面（`MISSED`案例多）。
    - **Tavily**：支持黄金答案的URL高度集中在排名第一的位置（占50%），当预获取支持存在时，其顶级结果策略很高效。
    - **Firecrawl**：关联于更广泛的无支持探索行为（`BLIND`模式），更多依赖页面获取后才发现的证据。
3.  **“表面矛盾”比率差异巨大**：片段中的矛盾信息与黄金证据的比率从Brave的0.92到Firecrawl的2.59不等，说明不同API的预获取表面噪声水平不同。
4.  **答案互补而非重叠**：仅有10个问题三者均回答正确，提供商的选择改变了哪些问题被成功解答，而非仅仅改变总个数。
5.  **传统检索指标不足**：许多失败案例表明，尽管正确答案文本在检索结果中可见，但代理仍未能有效利用。这证明了评估必须超越召回率，关注证据在决策表面的可见性、可信度和可操作性。

### 优点
- **视角新颖且深刻**：将搜索API从单纯的“检索器”重新定义为影响智能代理决策行为的“决策面”，这一概念提升了对搜索API评估的层次。
- **实验设计高度控制**：“冻结代理”方法论非常干净地剥离了混淆变量，确保观察到的所有差异均归因于搜索API表面本身，逻辑严谨。
- **细粒度的证据溯源**：建立了一套完整的逐URL预言机标注和决策四象限划分体系，可以从“证据何时以何种形式出现”的维度，精细地解释和量化不同提供商的差异。
- **实践指导性强**：提出的“表面矛盾比率”、`SMART`/`BLIND`/`MISSED`等分区及最终结论，为从业者如何根据自身代理的获取策略选择合适的搜索API提供了具体、可操作的分析工具。
- **审计与可复现性**：通过人工验证和语义审计对自动化流程进行了校准，并提供了完整的代码、配置和分析脚本，研究过程透明。

### 不足与局限
- **单一代言和预言机偏差风险**：所有结论基于单一的GPT-5.4代理和Kimi-K2.6预言机。如果更换答案模型、法官模型或系统提示，决策分区的分布和具体指标值可能会发生变化。
- **样本规模有限**：100个查询和3家提供商的规模适合做诊断性研究，但其结论的普适性（尤其是对更多提供商和更广泛任务类型）需要更大范围的验证。
- **观测性而非因果性分析**：实验观测了代理在不同表面下的行为，但并未通过干预手段（如主动交换排名、修改片段）来建立表面特征与代理行为之间的因果联系。
- **预获取片段信息量不对称**：Brave原生提供了更多片段文本，这被认为是其预获取支持优势的部分原因，这一产品本身的差异性虽然真实存在，但也为纯粹的排名质量对比引入了干扰。
- **支持度的下界估计**：由于只对代理实际看到的URL进行标注，而非对整个搜索引擎的返回结果进行全量评估，所测得的“预获取支持”和“矛盾”都是下限值，无法完全反映提供商的全貌。
