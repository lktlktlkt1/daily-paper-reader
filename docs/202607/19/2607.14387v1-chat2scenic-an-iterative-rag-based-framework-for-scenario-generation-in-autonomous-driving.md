## 摘要
验证自动驾驶系统需要多样化且符合法规的测试场景。在基于仿真的测试中，场景被定义为可执行的脚本。然而，从法规描述自动生成此类脚本仍然是一个公开挑战，现有方法面临基本权衡。检索-组装方法可实现合理的编译率，但缺乏可扩展性，而基于检索的全脚本生成则存在编译成功率低的问题。我们提出Chat2Scenic，这是首个生成领域特定语言（DSL）场景脚本的迭代式检索增强框架。具体来说，Chat2Scenic提供支持交互式场景精化的聊天机器人界面，并集成检索增强生成（RAG），将场景生成建立在法规知识和DSL语法基础上。此外，我们提出了一个用于场景生成的开放基准，包含来自NHTSA和联合国车辆法规等各种法规以及其他来源的123个场景。使用最新的大语言模型进行的广泛评估表明，Chat2Scenic实现了76.42%的编译成功率和58.17%的框架准确率，优于现有方法（检索组装为30.08%编译成功率、11.03%框架准确率，检索全脚本生成为16.26%编译成功率、10.86%框架准确率）。为促进未来研究，我们将代码开源在https://github.com/TUM-AVS/chat2scenic。

## Abstract
Validating autonomous driving systems requires diverse, regulation-compliant test scenarios. In simulation-based testing, scenarios are defined as executable scripts. Yet automatically generating such scripts from regulatory descriptions remains an open challenge, and existing approaches face fundamental trade-offs. Retrieval-assemble methods achieve reasonable compilation rates but lack scalability, whereas retrieval-based full-script generation suffers from low compilation success rates. We present Chat2Scenic, the first iterative retrieval-augmented framework to generate scenario scripts in Domain Specific Language (DSL). Specifically, Chat2Scenic provides a chatbot interface that supports interactive scenario refinement and integrates Retrieval-augmented Generation (RAG) to ground scenario generation in regulatory knowledge and DSL syntax. Furthermore, we propose an open benchmark for scenario generation comprising 123 scenarios from various regulations, including NHTSA and United Nations Vehicle Regulations, as well as other sources. Extensive evaluation with State-of-the-Art (SOTA) Large Language Models (LLMs) demonstrates that Chat2Scenic achieves 76.42% Compilation Success Rate (CSR) and 58.17% Framework Accuracy (FA), outperforming existing methods (Retrieval Assemble with 30.08% CSR, 11.03% FA and Retrieval full script generation with 16.26% CSR, 10.86% FA). To facilitate future research, we release our code as open source at https://github.com/TUM-AVS/chat2scenic.

### 2. 方法论：核心思想与关键技术

论文提出了 **Chat2Scenic**，一个基于**迭代式检索增强生成（RAG）**的场景生成框架。其核心思想是化整为零：将复杂的完整场景脚本生成任务，分解为一系列有序的、迭代的组件生成任务，并结合领域知识增强生成过程。

- **整体框架流程 (Algorithm 1)**：
    1.  **逻辑结构提取**：通过LLM解释器将用户输入转化为一个结构化的逻辑表示 `S = {G, R, E, O, T}`（即全局配置、空间关系、自车行为、目标对象集合、约束条件）。
    2.  **全局配置生成**：独立生成地图、天气等全局配置代码。
    3.  **迭代式组件生成**：按依赖顺序（`R -> E -> O` (迭代) `-> T`）逐个调用专门的生成器，每个生成器在生成当前组件时，会看到前面已生成的代码作为上下文，以确保组件间的兼容性。
    4.  **最终组装**：将所有生成的组件代码简单拼接，形成完整的可执行Scenic程序。

- **三大核心模块 (Fig. 2)**：
    - **交互模块 (Interactive Module)**：提供一个基于Gradio的聊天机器人界面。用户可以输入自然语言描述，系统通过LLM解释器将描述翻译为上述逻辑结构 `S_int`，并支持**用户通过聊天进行多轮交互式精化**。
    - **RAG模块 (RAG Module)**：包含两个数据库和两个检索器，为生成阶段注入领域知识。
        - **代码片段数据库**：存储从官方Scenic源码分解出的组件级代码单元及其自然语言描述。
        - **文档数据库**：存储Scenic文档和法规文件（如UN R152）的结构化切片。
        - **双检索器架构**：
            - **代码检索器**：根据组件描述进行语义搜索，召回最相似的代码片段作为示例。
            - **文档检索器**：使用混合检索（语义向量相似度 + BM25关键词匹配），并通过倒数秩融合（RRF）排序，召回相关文档片段，提供更广泛的语言和语法知识。
    - **生成模块 (Generation Module)**：接收逻辑结构和RAG模块提供的上下文，通过组合多种先进的提示工程技术指导LLM生成代码 (**Fig. 4**)。
        - **上下文提示**：在系统提示中注入领域知识、语法约束和类型系统。
        - **思维链**：为每个组件的生成设计特定的分步推理步骤（如“理解需求 → 检查上下文 → 选择类型…”）。
        - **上下文学习**：提供固定的正、负样例。
        - **检索增强上下文学习**：动态地从RAG模块中检索最相关的代码示例和文档片段，作为当前生成任务的上下文。这是框架泛化能力的核心。

### 6. 主要结论与发现

- **方法有效性**：Chat2Scenic所提出的“迭代式组件生成 + 高级提示技术”显著优于现有方法。在使用Gemini-3-Flash时，其76.42%的CSR和58.17%的FA远超ChatScene和NL2Scenic。
- **消融研究的关键发现**：
    - **上下文提示是基础**：在零样本（0% CSR）基础上加入上下文提示（CP），CSR跃升至12.20%。
    - **上下文学习是关键**：在CP基础上加入上下文学习（ICL），性能大幅提升。
    - **思维链与检索结合最佳**：CP + CoT + ICL + 代码检索增强（CodeICL）的组合（C11）达到了最优性能。
    - **代码示例比文档更有效**：添加代码检索（CodeICL）能显著提升生成效率和效果，而添加文档检索（DocICL）则因信息过于概括且不直接指导编码，反而可能引入噪声，未带来性能提升。
- **模型适用性发现**：框架性能高度依赖底层LLM的能力。指令优化型模型（如Gemini-3-Flash）可能比思维型模型（如Gemini-3-Pro）更适合此类需要遵循复杂指令和结构化提示的任务。当前的开源模型在复杂提示跟随和长代码生成上仍与闭源模型有显著差距。

### 7. 优点

- **创新的方法范式**：提出了“迭代式组件生成”这一新范式，巧妙地结合了检索-组装方法（高成功率）和直接生成方法（高灵活性）的优点，并规避了它们的缺点。
- **深度融合的RAG设计**：框架内建了专门设计的双数据库（代码、文档）和双检索器，将RAG的应用从简单的知识补充提升到了可动态增强提示（检索增强的上下文学习）的层面。
- **强大的交互性设计**：通过聊天界面支持用户对场景进行多轮精化，使框架更具实用价值，而不仅仅是一个端到端的自动生成器。
- **扎实的评估体系**：自主构建了首个公开的法规型场景基准，并设计了包含编译成功率和分层语义准确率的综合评估指标（FA），为后续研究提供了有价值的基准和评价标准。
- **良好的可复现性**：完全基于提示工程，无需微调，且代码和基准均已开源。

### 8. 不足与局限

- **场景准确性评估的主观性**：场景质量的评估依赖人工对语义层进行打分，不可避免地引入主观性，且难以大规模复现。
- **基准的有限规模与多样性**：尽管基准来源多样，但123个场景的总体规模仍然有限，可能无法完全覆盖真实世界中复杂的“长尾”问题和DSL中的复杂组合。
- **对底层LLM的强依赖性**：框架本身是LLM能力的一个放大器，但无法弥补骨干模型基础能力的不足。其对闭源高端模型的依赖可能带来成本和稳定性风险。
- **效率问题**：迭代式生成虽然成功率高，但平均生成时间较长（最优配置下约222秒），离实时交互式场景构建仍有距离。
- **领域局限性**：当前框架是为Scenic DSL和CARLA仿真器定制的，迁移至其他DSL（如OpenSCENARIO）或仿真器可能需要重新设计逻辑结构模板和数据库。
